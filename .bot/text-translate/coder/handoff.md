## One-line scope

Kill `tstring` (done), and replace it with a **`translate` axis on `text`**:
user-facing prose is marked translatable on its `text` type, and a `text`
renders through a translator service (`app.translate`) at the output edge.

## Part B — TO DESIGN + BUILD (the feature)

### Settled with Ingi

- **Locale = the sender's / emitting edge** (decision A, resolved in the origin
  doc). Translation fires at the actor that produces the text; the flag is
  **`.pr`-only**; the runtime channel wire stays a bare string. The `.pr` `type`
  object is exactly where the `.pr`-only flag belongs (it rides next to `kind`).

- **`text.Translate` — a per-value marker, type `translate?`:**
  ```
  null/absent → don't translate
  true        → translate { Language = null }   // ambient: app.translate picks the language
  {language}  → translate { Language = "fr" }    // forced
  ```
  The LLM chooses `true` vs `{language:…}` on the wire; the runtime normalises both
  to a `translate` instance (`true` = a `translate` with no language). One stored
  representation — `true` is not a separate thing, it's a `translate` with an
  unspecified language.

- **`translate` type:** `{ language? }`. **No `key` property** — the text's *value*
  IS the translation key (`app.translate[this.value]`). (`context`/`note` for
  homograph disambiguation, and plural-category, are candidates for later — start
  language-only.)

- **`app.translate` — the translator service on `app`.** A `text` reaches it via
  its `Context → App` (Context rides on the Data/value, as `text.Value(asking)`
  already does). It owns the **ambient target language** (Ingi's "defined somewhere
  else" — to be specified by him). Lookup is keyed on the source text:
  `app.translate[this.value]`.

- **Render** (inside `text.Value()`), order matters — translate the **authored
  template** first (holes intact), THEN fill holes:
  ```
  text "Hey %name%", Translate = {language:"fr"}
    → Translate != null  → reach app.translate via Context
    → app.translate["Hey %name%"]  (forced fr)        → "Bonjour %name%"
    → fill holes                                       → "Bonjour Ingi"
  ```
  The lookup key is the authored template, not the rendered string.

- **`(lang:fr)` authoring syntax** — `- write out "Hey %name%"(lang:fr)` — treated
  as an **LLM-interpreted authoring hint** (no new parser grammar), the LLM emits
  the structured `translate:{language:"fr"}`. Confirm.

### The load-bearing piece — generic type-carried properties

Ingi's framing: *"each type can have a property; discover it at `.pr` read time and
set it."* This is bigger than `translate` and is the real foundation:

> The `.pr` `type` object becomes a **property initializer** for the value.
> `name` picks the type; **every other key** (`kind`, `strict`, `translate`, …) is
> an init-property discovered on that type and deserialized via **its own type's
> reader** (recursive — `translate:{language:"fr"}` deserializes through the
> `translate` type), then stamped onto the value **at construction** (born-typed —
> not a post-hoc mutation).

Today `type/this.json.cs` hardcodes `name`/`kind`/`strict` and `reader.Skip()`s the
rest, and `type.Create(name, kind, strict)` is the construction surface. Generalising
to a property bag is a **read-path refactor**, and it should **fold `kind`/`strict`
into the generic mechanism** rather than keep them special-cased.

### Open questions for the architect

1. **Generic mechanism shape.** Reflect the value type's init-only properties,
   match `.pr` `type` keys to them, deserialize each via its own type reader, and
   pass the bag at construction? How does this interact with the born-typed /
   deserialize-flow model on `compare-redesign` (the type's reader owns
   `object → item`)? Does `type.@this` grow a property bag, or does the converter
   hand the bag straight to the value's reader?

2. **Marker vs service — relationship & naming.** Is `text.Translate` the small
   `translate` type `{language?}` and `app.translate` a *separate* service, or is
   `app.translate` itself the `translate` type? This decides whether `app.translate`
   follows the `app.X`-is-a-collection convention or is a no-`.current` service
   (like `app.module`). (Coder leans: small `translate` marker type + `app.translate`
   service.)

3. **Ambient language home.** Does `app.translate` own the default target language,
   or does it read it from actor/context/settings? Ties to Ingi's "defined somewhere
   else" — he will specify. Note it stays consistent with decision A: the emitting
   edge translates, using the forced `(lang)` or the ambient default.

4. **`translate` type surface.** Confirm `{ language? }` to start; decide whether
   `context`/`note` (homograph disambiguation) and plural-category join now or later.

5. **Build/teach gating.** The teaching already emits `translate`; until Part B
   lands the converter drops it. Fine as a forward-correct `.pr` shape, or should
   the teaching hold until the mechanism exists? (Coder view: forward-correct is OK
   — the flag persists harmlessly and a later `plang build` doesn't need re-teaching.)

### Demolition that completes when Part B lands

- The `translate` type (`app.type.translate`) + `text.Translate` property.
- `app.translate` service.
- Generic type-property read mechanism (or a scoped version if the architect
  prefers translate-only first).
- A `plang build` sweep regenerates all `.pr` (clears the stale `type:"tstring"`
  and emits the new `type:{name:"text", translate:…}` where prose is detected).
