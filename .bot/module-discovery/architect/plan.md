# Stage 4 — module-discovery: the collection at `app.module`, teaching as templates

Settled with Ingi 2026-07-15. Supersedes the shape parts of `coder/stage4-plan-seed.md` (the seed's decomposition survives; the model corrections below override it).

> **You (coder/test-designer) own the final code and test shape.** Shapes below pin intent, not lines. Every `file:line` was checked against `c1e4f8e10`; re-verify at implementation time.

## Why

`module.@this` is the last god-object: it owns action dispatch AND teaching (`Describe()` — imperative reflection assembling prompt text in C#, `StepActions` string blobs, `BuildTypeEntries(modules)` feeding the type views). One way for each thing: **selection via the module collection, teaching via element structure + Fluid templates, prose via the elements' own markdown.** Deleting the three teaching members unblocks Stage 5's sweep (3 of its remaining `[Obsolete]` marks), retires `goal.getTypes` (the string-typed shadow of the type system), and makes the builder prompt user-editable. 4f applies the same presentation rule to the test report and deletes the bespoke junit serializer (yagni).

## The settled model

1. **`module` and `action` are HOSTS, not plang types** (Ingi ruled; item ⟺ ICreate decides it — they are never authored, never created from values). Carried as `clr(module)`/`clr(action)`, navigated by the reflection kind, rendered through `[Out]` faces, read by Fluid directly. Same family as `app`/`goal`. The catalog action is **the same `action` host the `.pr` uses, at class-level zoom** (parent plan's ruling) — trace item: verify the `.pr` action's step-level state doesn't leak into the class-zoom face; if the two zooms fight, surface it, don't fork the type.
2. **The collection is the node: `app.module`** — `app/module/list/this.cs` (`app.module.list.@this`), hand-rolled like `goal.list`, mutable (`code.load` adds at runtime). Selection `app.module["http"]`; dispatch reads through the element (`module.Actions["request"]` answers the handler type). Registry = selection + lifecycle; behavior on the element. The old `module.@this` dissolves into collection + elements over 4a–4e.
3. **The property is lowercase: `app.module`** — Ingi killed the "properties on `app.@this` stay PascalCase" rule (singular + lowercase, both). This plan uses lowercase for the NEW node; the global rename of existing properties (`.Cache`, `.Goal`, `.FileSystem`, …) is its own mechanical pass, NOT this branch (logged in the close-out; CLAUDE.md proposal filed in `.bot/module-discovery/claude-md-proposals.md`).
4. **One handler walk.** The collection's population walk over handler classes does every per-handler job in one pass: builds the module/action elements AND registers choice closed sets + their `Reader<T>`s. `RegisterModuleChoiceTypes` (verb+noun obpv, `type/list/this.cs:483`, called from `app/this.cs:307`) dissolves into that walk — no public compound-name method survives.
5. **Reflection once, at the leaf — and the row is a `property` host, not a bare `type`** (corrected in review: a param row is type ENTITY + nullability + default + the `%var%` marker; `list<type>` under-modeled it). `action.Properties : list<property>` is the ONLY reflection site — lazy, cached on the element, with exactly `Describe()`'s current filters (capability interfaces, `[Code]`, `EqualityContract`), unwrapping `Data<T>`/`[Code] T`/`Nullable<T>` to the plang `type` entity. Consumers read `type.Name`, never a `System.Type`, never `GetTypeName` at a call site.
5b. **Surfaces return the NATIVE plang list** (Ingi ruled): `app.module.list`, `module.Actions`, `action.Properties` all answer `app.type.item.list.@this` — filterable by the list module with zero conversion. **INVARIANT (coder's #1, ruled):** every dissolved catalog surface — including the `app.module`/`app.type.list` navigation that replaces `build.actions`/`build.types` — hands back the NATIVE list, never a clr-wrapped host collection: `list.where` gates on subject type (`list/where.cs:36`) and a clr host falls to its apex error → a silently empty filter at build time. Spike acceptance for leg (e) is therefore `where` over the REAL catalog surface, not a synthetic `item.list`. No extra loop vs `IEnumerable` (the one materialization moves into the owner; a lazy iterator would have ridden as `clr(iterator)` anyway). **Elements cached, wrappers fresh**: module/action elements + reflection rows + prose cache on the element, invalidated by the registry's own mutations (`RegisterType`/`Register`/`Remove`/`Clear`); the list wrapper mints fresh per ask over the same cached elements. Catalog-zoom actions get an internal `Context` stamped at mint (null on `.pr` instances).
5c. **4a is a SPLIT, not a parallel build** (found writing the code): today's `module/this.cs` already IS the collection (`_modules` index, Discover, Register, lifecycle, `GetCodeGenerated`) — it relocates to `module/list/this.cs` and sheds behavior (teaching dies/dissolves; per-action queries move onto the action element); `module/this.cs` is freed for the element. `GetChannelInventory` dies outright (a middleman over `actor.Channel`). Choice registration fires on App-attach + inline on register — fixing the latent gap where `code.load` modules never got their choice sets registered.
6. **Teaching is ALREADY ~90% templates** (coder's comment round corrected the plan here — `Describe()` builds STRUCTURE; four Fluid templates render all text: `os/system/actions/v2/summary.planner.md`, `v2/summary.md`, `os/system/builder/llm/templates/stepActionDetails.template`, v1 `summary.md`). The only C# text composition left is the per-param `desc` glue (`module/this.cs:341-352` + `FormatDefault` + `IsVariableNameSlot`) — it moves INTO the templates' param loops (facts on the `property` row; text in the template). Don't hunt for more C# text assembly; there isn't more.
6b. **Template reshape (Ingi, this round):** the two SUMMARY templates flip to iterating module elements (tree) — `{{ module.Name }} — {{ module.Description }}`, actions inside; the flat `map|uniq|sort` + `where:"Module"` + `first.ModuleDescription` grouping gymnastics die (it's a MODULE's description — just `Description`, on the module). `stepActionDetails` stays fed by a FILTERED flat action list: `Start.goal:16`'s C# subset filter becomes `- where %actions% Name in %planStep.actions%, write to %actions%` — the EXISTING `list.where` + the Operator registry's `"in"`; the template's nested dedup-match loop (`stepActionDetails.template:14-15`) dies. The catalog action gains **`Name`** (`"file.read"` — templates' `{{ a.Module }}.{{ a.ActionName }}` collapses to `{{ a.Name }}`). NO new query machinery: the list module already has where/sort/group/first/any/count/unique; the lazy/deferred (yield-like) upgrade is logged as its own future piece, nothing here depends on it.
6c. **`build.actions` and `build.types` dissolve into navigation + templates.** Full catalog: the goals reach the collection natively (`%!app...%` → `app.module` → `.list`). Type vocabulary for the LLM: render a template over the TYPE entities (`app.type.list` — verify/add the collection's enumeration door; the Schema view's C#-side assembly goes the same way as Describe's). Whether thin actions survive as conveniences is coder's call at 4e — no C# text assembly either way.
7. **Prose lives on the elements.** `MarkdownTeaching` (static helper class + compound name, both obpv) dissolves: `module`/`action` own loading their prose (`module.Description`, `action.Description`/`.Notes`/`.Examples` read `os/system/modules/<module>/…md` themselves; the orphan scan becomes the collection's own audit at population). Nothing central remains.
8. **Template format = the extension.** The error-page precedent (`os/system/error/<code>.<ext>`) generalizes: same stem, extension picks the format — `report.txt` beside `report.json`; `ui.render` resolves stem + requested format; mime from the extension via `app.Format`. Inside a `.json` template, values embed via the Fluid `json` filter, and that filter MUST route through our json writer — the one-JSON-producer rule; verify what Fluid's stock filter does and rewire if it serializes on its own.
