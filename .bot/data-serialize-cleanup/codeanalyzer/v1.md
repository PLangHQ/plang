## F1 — `Stream.Length` throws on non-seekable streams; gets swallowed as a deserialize error (HIGH)

**Files:**
- `PLang/app/channels/serializers/serializer/plang/this.cs:112` (`DeserializeAsync`)
- `PLang/app/channels/serializers/serializer/plang/this.cs:127` (`DeserializeAsync<T>`)
- `PLang/app/channels/serializers/serializer/Json.cs:96` (`DeserializeAsync`)
- `PLang/app/channels/serializers/serializer/Json.cs:111` (`DeserializeAsync<T>`)

```csharp
if (stream.Length == 0) return global::app.data.@this.Ok();
```

`Stream.Length` throws `NotSupportedException` for non-seekable streams (network sockets, pipes, HTTP response bodies, anything wrapped in `GZipStream` for read). The catch clause includes `NotSupportedException`, so the exception is silently converted to a `PlangDeserializeError`/`JsonDeserializeError` 400 — every read from a non-seekable stream becomes "deserialize failed."

This is invisible in unit tests (always seekable `MemoryStream`) and the integration cuts the test-designer wrote (`PLang.Tests/App/Serialization/*`) all use `MemoryStream` per the contract stubs. The bug only surfaces when the merged plang serializer is wired to a non-seekable channel — exactly the case Stage 1's input-tightening was supposed to unify.

**Fix:** guard the length check, or drop it and let an empty payload surface a JsonException naturally:
```csharp
if (stream.CanSeek && stream.Length == 0) return global::app.data.@this.Ok();
```
Same fix at all four sites.

## F2 — `WireJsonConverter.MarkOuterForHash` loses tracking under nested hash of the same Data (MEDIUM)

**File:** `PLang/app/data/WireJsonConverter.cs:39-61`

`_hashOuter` is an `AsyncLocal<HashSet<@this>>` with `ReferenceEqualityComparer`. The Scope removes the Data on dispose and nulls the field when the set is empty. Sequence to consider — two nested `Hash` calls on the same Data instance D:

1. Outer scope adds D → set = {D}
2. Inner scope adds D → `HashSet.Add` is no-op (already present), set still {D}
3. Inner scope disposes → set.Remove(D), set empty, `_hashOuter.Value = null`
4. Outer scope continues serialising → `IsHashOuter(D)` returns false → converter calls `EnsureSigned()` on D → recursive Hash call → loop

Nested Hash on the same Data is realistic: `signing.verify` re-hashes the inner Data while the outer Verify is mid-serialise; a manual `crypto.Hash` action invoked from inside a goal whose own result is being signed; future replay tests.

**Fix:** ref-count per Data instead of set membership, e.g. `Dictionary<@this, int>` with increment/decrement, or have `Scope` capture whether *it* was the one that added D and only remove on the matching dispose.

## F3 — Sync-over-async in `Compress` / `Decompress` / `EnsureSigned` (MEDIUM)

**File:** `PLang/app/data/this.Transport.cs:73, 121, 184`

```csharp
var result = _context.App.RunAction(action, _context).GetAwaiter().GetResult();   // EnsureSigned
serializer.SerializeAsync(ms, this).GetAwaiter().GetResult();                      // Compress
var deser = serializer.DeserializeAsync(ms).GetAwaiter().GetResult();              // Decompress
```

Three `.GetAwaiter().GetResult()` calls from sync methods that are exclusively called from async contexts:

- `EnsureSigned` is called by `WireJsonConverter.Write` which runs inside `JsonSerializer.SerializeAsync` — a sync `Write` is the contract, but the work it triggers (`RunAction`) is async.
- `Compress`/`Decompress` are called by `variable.compress.Run()` / `variable.decompress.Run()` — both action handlers that return `Task<Data>` and could trivially be `async Task`.

Memory `analysis_patterns.md` flags this as a recurring finding: deadlock risk under sync contexts, and 3× the allocation cost. The action-handler sites are pure quality (`return Task.FromResult(await target.CompressAsync())` is one keystroke away). `EnsureSigned` is harder — JsonConverter.Write is sync by STJ contract — but the *real* fix is to make signing a pre-walk async pass that EnsureSigns the whole graph before the sync serialise begins, leaving the converter signature-aware but no longer async-bound. Worth tracking even if not landed this branch.

## F4 — Hash's serializer resolution uses `as` cast that silently falls back (MEDIUM)

**File:** `PLang/app/modules/crypto/code/Default.cs:38-40`

```csharp
var serializer = action.Context?.Actor?.Channels.Serializers.GetByType("application/plang")
                 as global::app.channels.serializers.serializer.plang.@this
                 ?? _fallbackPlang;
```

If a consumer registers a different `ISerializer` for `application/plang` MIME (custom transport serializer, test double, future format pivot), the `as` cast returns null and the `??` falls through to the static context-less `_fallbackPlang`. The hash silently uses different bytes than the registered serializer would have emitted — wire-bytes and hash-bytes diverge, which is exactly the Stage-2 bug this stage was meant to fix.

The cast is needed for `OutboundOptions` (internal property exposed on the concrete type, not on `ISerializer`). The right shape is either to expose `OutboundOptions` on the interface, or to fail loud when the registered serializer isn't the canonical one:
```csharp
var registered = action.Context?.Actor?.Channels.Serializers.GetByType("application/plang");
if (registered != null && registered is not global::app.channels.serializers.serializer.plang.@this)
    return data.@this<byte[]>.FromError(new ServiceError(
        "Registered application/plang serializer is not the canonical plang serializer; hash would diverge from wire", "SerializerMismatch", 500));
var serializer = (registered as global::app.channels.serializers.serializer.plang.@this) ?? _fallbackPlang;
```

## F5 — `Variable.Resolve` allows negation prefix + property suffix to silently produce wrong lookup (MEDIUM)

**File:** `PLang/app/variables/Variable.cs:71-115`

`%!flag%` is negation; `%x!cost%` is Property access. The parser handles each in isolation. But `%!x!cost%` is parsed as:

- `hasNegationPrefix = true` (line 71)
- `body = "x!cost"` (line 72)
- `bangsInHead = 1` → falls into the property branch (line 100)
- `name = "!x"`, `property = "cost"` (line 103-104)

Then `variable.set` does `Context.Variables.Get("!x")` (set.cs:66), which finds nothing — `!x` is not a registered variable name. No `IsMalformed` flag is set, so the user gets a `VariableNotFound` for "!x" rather than a syntax error telling them negation and property-suffix can't coexist.

Either reject this shape at parse time (`bangsInHead == 1 && hasNegationPrefix → malformed`), or document that property-write on a negated reference is unsupported and fail with a typed error.

## F6 — `Properties.Get<T>` bare catch around `Convert.ChangeType` (LOW)

**File:** `PLang/app/data/Properties.cs:86-87`

```csharp
try { return (T)Convert.ChangeType(v, typeof(T)); }
catch { return default; }
```

The bare catch swallows `OutOfMemoryException`, `StackOverflowException`, `ThreadAbortException`, `NullReferenceException`. The intended catches are `InvalidCastException`, `FormatException`, `OverflowException`. Cheap fix:
```csharp
catch (Exception ex) when (ex is InvalidCastException or FormatException or OverflowException) { return default; }
```

Memory `analysis_patterns.md`: "Bare catches are the #1 reliable finding" — applies even on coercion helpers.

## F9 — `Properties.EnsureSupportedValue` doesn't recursively validate dict/list contents (LOW)

**File:** `PLang/app/data/Properties.cs:115`

```csharp
if (value is IDictionary<string, object?> or IEnumerable<object?>) return;
```

A `List<object?> { new SomeRandomDomainObject() }` passes the check, gets stored, then explodes at wire-emit when the WireJsonConverter walks Properties and STJ encounters the un-serialisable child. The error then surfaces at the wire boundary, not at the Set site that introduced the bad value — defeating the EnsureSupportedValue gate's purpose (early, clear, attribute-the-blame-to-the-caller).

Either recursively validate, or document the gate as "best-effort top-level only" and accept the late surface. Recursive validation is cheap (few-element bags); recommend doing it.

## F10 — `_fallbackPlang` static singletons duplicated across files (LOW)

**Files:**
- `PLang/app/data/this.Transport.cs:26`
- `PLang/app/modules/crypto/code/Default.cs:21`

Both files declare a private static `_fallbackPlang` field, each constructing a fresh `plang.@this()`. Two singletons of "the same thing" with the same xmldoc. If the fallback construction ever needs to change (additional converter, options tweak), both have to change in lockstep — exactly the OBP smell #5 the project CLAUDE.md describes ("Allocate-here / mutate-there"). Recommend a single `plang.@this.ContextLessFallback` static (or method) that both sites call.

## F11 — `Unwrap` mutates the inner Data's Context as a side effect of a read (LOW)

**File:** `PLang/app/data/this.Transport.cs:217-222`

```csharp
public @this Unwrap()
{
    if (Value is @this inner)
    {
        inner.Context = _context;
        return inner;
    }
    return this;
}
```

`Unwrap` is conceptually a read (strip the outer, return what's inside). It writes to `inner.Context` — and `inner` is a reference that may live in someone else's value graph (the original outer's `Value`). Calling Unwrap from two different Contexts thrashes `inner.Context` to whichever was last. `Decompress` (line 194) and `Compress`/`Wrap` (lines 96, 125) do the same Context assignment, but they construct fresh outers so the assignment isn't a mutation of a shared graph.

If two consumers Unwrap the same outer in different actor contexts (forwarding case the architect plan motivates), the inner's Context becomes whichever Unwrap ran last. For Context that's just a "default for further calls" hint, this is probably fine in practice — but it's worth a comment, and worth thinking about whether Unwrap should return a Context-rebound *clone* rather than mutating the shared inner.
