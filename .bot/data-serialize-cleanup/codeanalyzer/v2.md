## Critical — all resolved

### F1 — `Stream.Length` on non-seekable streams

**Fixed.** All four sites guarded with `stream.CanSeek && stream.Length == 0` (`plang/this.cs:121`, `:136`; `Json.cs:96`, `:111`). The empty-`MemoryStream` shortcut still runs for the test-fixture path; non-seekable streams (HTTP response bodies, sockets, `GZipStream` for read) fall through to the real deserialise path instead of getting silently surfaced as `JsonDeserializeError`/`PlangDeserializeError`.

### F2 — `MarkOuterForHash` lost tracking under nested same-Data hashes

**Fixed.** `HashSet<@this>` swapped for `Dictionary<@this, int>` keyed by `ReferenceEqualityComparer`. `MarkOuterForHash` increments; `Scope.Dispose` decrements and only removes the key when the count hits zero; `_hashOuter.Value` is nulled only when the whole dictionary empties. The nested-Verify and chained-Hash scenarios from v1 now compose. Latent infinite-recursion path closed.

### F3 — Sync-over-async in Compress/Decompress

**Fixed at the action-handler boundary.** New `CompressAsync(CancellationToken)` and `DecompressAsync(CancellationToken)` on `data.@this`; `variable.compress` and `variable.decompress` handlers switched to `async Task<data.@this>` and `await` the async variants. The sync `Compress()`/`Decompress()` wrappers were kept (delegating to `.GetAwaiter().GetResult()`) for non-async C# composition sites — acceptable as documented in the xmldoc on each wrapper. The `EnsureSigned` site inside `WireJsonConverter.Write` remains sync-over-async by STJ contract; that's a deeper redesign (pre-walk async pass) and was correctly scoped out of this branch.

### F4 — Silent serializer-type fallback in `crypto.Hash`

**Fixed.** `crypto/Default.cs:38-45` now reads the registered `application/plang` serializer, checks for type mismatch, and returns a typed `SerializerMismatch` 500 error when the registration isn't the canonical `plang.@this`. The wire-vs-hash divergence the v1 finding warned about (a custom transport serializer producing different bytes than the hash) now surfaces as a loud failure instead of a silent canonicalization regression of the Stage 2 fix.

### F5 — `%!x!cost%` silently produced wrong lookup

**Fixed.** `Variable.Resolve` (`Variable.cs:99-118`) now branches: in the single-bang-in-head case, if `hasNegationPrefix` is true the parser flags `IsMalformed = true` and keeps the full body on Name; only the non-negated case extracts a property. The `VariableResolveTest.Resolve_NegationPlusProperty_*` test was renamed and rewritten to assert `IsMalformed = true` and `Property = null`. Write paths through `variable.set` now surface `InvalidVariableReference` 400 instead of `VariableNotFound` on the synthetic `"!x"` name.

### F9 — non-recursive `EnsureSupportedValue`

**Tradeoff-documented (accept).** Coder kept the gate best-effort top-level and added a paragraph explaining why: real domain-object containers (`List<LlmMessage>` and similar) flow through Properties today, and recursive validation would reject them. The right tightening lives at the upstream producers (convert to dict/list-of-primitives before storing) and is tracked separately. The gate still enforces the load-bearing Stage 4 invariant — `Data` instances cannot enter Properties (explicit check, dedicated error message). Conscious deferral with rationale on record.

Side note: the gate actually *widened* in this commit — from `IEnumerable<object?>`/`IDictionary<string,object?>` to `System.Collections.IEnumerable`/`IDictionary`. This is consistent with the documented intent (accept all reasonable containers, validate at producer), but worth noting that the check is now structurally permissive of any IEnumerable. The Data-instance gate runs before the IEnumerable check, so `IEnumerable<Data>` still hits the Data-rejection path on the first element only if a producer happens to call EnsureSupportedValue per-element — at the top level a `List<Data>` would currently slip through. If a future test demands strict primitive-only containers, the producers are the place to fix it.

### F10 — duplicated `_fallbackPlang` singletons

**Fixed.** Single source of truth on `plang.@this.ContextLessFallback` (public static readonly). Both `Transport.CompressAsync`/`DecompressAsync` and `crypto/Default.Hash` route through it. The local `_fallbackPlang` in `crypto/Default.cs` is now a property expression that forwards to the canonical static. No drift surface.

Initialization order check: `ContextLessFallback = new plang.@this()` invokes the parameterless ctor → builds `JsonSerializerOptions` with `WireJsonConverter`, `Transport.ForOutbound`, `path.JsonConverter()`. None of those reach back into `plang.@this` statics. Acyclic, safe.

### F11 — `Unwrap` mutates `inner.Context` on read

**Tradeoff-documented (accept).** Coder added a six-line `<para>` to the `Unwrap` xmldoc acknowledging the shared-reference side effect, naming the race condition (two consumers in different actor scopes Unwrapping the same outer), and noting the future tightening (clone-and-rebind when forwarding starts requiring isolated provenance). In current usage `Context` is a "default for further calls" hint rather than identity, so the race is benign. The deferral is on record where any future reader will see it.
