# v1 Review Summary

Creator reviewed the 12 security findings and provided fundamental corrections to the threat model:

1. **#4 (system variable writes) and #5 (ObjectNavigator reflection) are by design.** PLang's security model is user-sovereign: the user owns their software. .pr files run in a trusted environment. Writing to `!fileSystem` is the user's prerogative, same as `Environment.Exit()` in C#. These should not be flagged as injection vulnerabilities.

2. **#9 (Signature/Verified) is a real problem, but fix direction matters.** Signatures are PLang's trust boundary. Once data is cryptographically verified, everything inside runs with full trust. The fix: `Verified` should NOT be a settable bool — it should be `{ get; private set; }` or computed from Signature via a `VerifySignature()` method. Flag anything that lets data claim "verified" without actual cryptographic proof.

3. **#8 (Newtonsoft namespace spoofing) is low practical risk.** If `library.load` gives code execution, namespace spoofing adds nothing new. Worth hardening (check assembly name) but not blocking.

4. **#1-3, #7, #10 (recursion) are real.** These are DoS vectors at trust boundaries where untrusted data enters. Should be fixed.

## Key Insight: PLang's Trust Model

- **Trust boundary = cryptographic signatures.** Not .pr files, not Variables writes, not reflection access.
- **Inside the wall = full trust.** Once data is verified, it runs with full power.
- **User-sovereign.** The user owns their code. Don't treat user actions as attacks.
- **.pr files are trusted.** They're generated by the builder from user code and run locally.
- **Untrusted data enters via transport** — channels, decompressed payloads, external inputs. THAT is where security matters.
