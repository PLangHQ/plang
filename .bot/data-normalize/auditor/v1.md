## What this is

Branch lands structural normalization on `app.data.@this`:

- `this.Normalize.cs` walks any C# value into a uniform tree
  (`primitive | byte[] | Data | List<>`), bounded by a 128-depth cap +
  reference-cycle visited-set.
- `this.Reconstruct.cs` is the reverse walker, dispatched per-target via
  `FromNormalized(Data, Context)` discovery and a special `path.@this` hook.
- `IWriter` + `json.Writer` give format encoders a no-reflection emission
  surface; JSON is the first impl.
- `filters/Tagged.cs` decides which CLR properties ship per `View`
  (`Out` / `Store` / `Debug`); `View.Store` is wired into `Sqlite.cs` so
  `[Sensitive, Store]` (Identity.PrivateKey) round-trips local persistence
  while staying excluded from the wire.
- `WireJsonConverter.cs` is renamed to `Wire.cs`; outermost Properties
  walk + nested Properties walk both thread `View` through `NormalizeValue`,
  closing the V1 hard-coded-`View.Out` site that codeanalyzer v3 fixed.

## Cross-file tracing — what I checked

The auditor's job is to follow data across boundaries. Three traces:

1. **Outbound Store-path (Sqlite.Set → Wire.Write → sign-if-missing).**
   `Sqlite.Set` → `plang.Store(data)` → JSON options with `Wire(View=Store)`
   → `Wire.Write` fires `EnsureInnerSigned(data.Value)` then `EnsureSigned()`
   on the outer. `EnsureSigned` dispatches through the signing action which
   re-serialises via `OutboundOptions` (View=Out) under `MarkOuterForHash`.
   Signature is therefore computed over the **Out** canonical bytes
   regardless of which view is being persisted — Sqlite stores the
   Store-view bytes alongside the Out-canonical signature. Symmetric on
   read: any later `signing.verify` re-canonicalises via Out and the
   signature matches. **No drift.**

2. **Inner-Data view propagation (the V1 site).** `Wire.Write` calls
   `data.Normalize(View)` for the outer value slot; the normalized tree can
   contain copies of inner `@this` nodes (see `this.Normalize.cs:73-87`).
   `json.Writer.Value` dispatches `case @this nested` → `BeginRecord` →
   `Value(nested.Value)` → `EndRecord(nested)`. `EndRecord` walks
   `record.Properties` using the writer's `_view`, not a hard-coded
   `View.Out`. Outer Properties go through `Wire.Write` directly (same view
   threaded into `NormalizeValue`). **Both sites view-consistent;**
   `CanonicalizationTests.StoreView_PropagatesIntoInnerDataProperties…`
   pins the path with a tampered-bytes assertion both ways (Out excludes
   `PRIV-must-persist`, Store keeps it).

3. **Reconstruction error visibility.** Every failure path in
   `this.Reconstruct.cs` throws `NormalizeException` with a typed key
   (`NormalizeReconstructFailed`, `NormalizeNoReconstructionStrategy`,
   `NormalizeMissingRelative`, `NormalizeContextRequired`,
   `NormalizeUnexpectedLeafType`, `NormalizeGetterThrew`,
   `NormalizeCycleDetected`, `NormalizeMaxDepthExceeded`). No silent
   swallowing, no `catch { }`, no bare `Console.WriteLine`. Getter
   exceptions are re-thrown with the original `ex` attached. **Error
   propagation discipline holds.**

## Findings

### Minor 1 — `Reconstruct` positional-ctor: missing required params silently default

`PLang/app/data/this.Reconstruct.cs:155-166`

```csharp
for (int i = 0; i < pars.Length; i++)
{
    var pname = pars[i].Name ?? "";
    if (byName.TryGetValue(pname.ToLowerInvariant(), out var c))
        args[i] = Walk(c.Value, pars[i].ParameterType, ctx);
    else if (pars[i].HasDefaultValue)
        args[i] = pars[i].DefaultValue;
    else
        args[i] = pars[i].ParameterType.IsValueType
            ? Activator.CreateInstance(pars[i].ParameterType)
            : null;
}
```

A positional record with a required reference parameter (no default,
no matching child in the normalized tree) gets `null` silently; a required
value parameter gets `default(T)`. The ctor then either accepts the
defaulted args (producing a half-populated record that may invariant-violate
downstream) or throws a generic `TargetInvocationException` caught at
line 171 and wrapped as `NormalizeReconstructFailed` with the ctor's own
message.

Surface area is narrow today — none of the inventoried domain types
(`Identity`, `setting`, `Variable`, `path.@this`, `http.Response`) use
positional ctors with required non-defaulted params. But the failure mode
is exactly the OBP "silent default population" smell — a future positional
record added to the wire vocabulary won't get a clear
`NormalizeMissingRequiredField` error; it'll either work-by-accident or
surface as a confusing ctor-internal exception.

**Fix shape** (when a use case arrives — not blocking): synthesize
`NormalizeException("NormalizeMissingRequiredField", ...)` when
`pars[i]` has no default, is not nullable, and `byName` lacks the key.

### Minor 2 — `Reconstruct` positional-ctor comment vs. behaviour

`PLang/app/data/this.Reconstruct.cs:151-152`

```csharp
// Prefer the ctor whose parameter names all match available children.
var best = ctors.OrderByDescending(c => c.GetParameters().Length).First();
```

The comment says "Prefer the ctor whose parameter names all match
available children" but the code just picks the **longest** ctor
unconditionally — no name-matching scoring. Today every C# record gets
exactly one public primary ctor + a copy ctor, so longest-wins lands on
the primary ctor reliably. If a type ever declares a secondary ctor with
more parameters than the primary (e.g., for testability), this picks the
wrong one.

**Fix shape:** either align the comment to "longest public ctor (records'
primary ctor in practice)" or actually implement the name-matching the
comment claims.

## What I didn't flag (and why)

- **`Tagged` transparent fallback for untagged types.** Architect-intentional
  (verb.@this and friends ride on this), captured by security v1 as a
  standing footgun. Not adding noise.
- **`json.Writer.Value` IEnumerable fallback passes `BeginArray(-1)`.**
  The `IWriter.BeginArray(int count)` contract is implicit (JSON ignores
  count, protobuf would not). The current call site only reaches that
  branch for IEnumerables Normalize couldn't materialise — in practice
  `Normalize` always produces `List<object?>` or `List<@this>` whose
  `.Count` is known. The bare `IEnumerable` arm is a defensive belt; the
  count-vs-known-count tension is a problem for the second writer impl
  (when it ships), not this branch.
- **`EnsureInnerSigned` doesn't traverse into domain objects holding
  inner Datas.** No current `[Out]`/`[Store]` property anywhere in the
  inventoried domain types is typed `Data` or contains Data inside a
  non-collection field. The walker covers @this / IDictionary / IEnumerable,
  which is the full reachable graph today. Captured to security memory
  for the day someone adds a Data-bearing domain type.
- **`Reconstruct<T>` has no depth bound on its own (only when reached via
  `Wire.Read`'s 64-depth gate).** Already acknowledged in the type's
  doc-comment + security v1's notes — T is caller-controlled and the
  in-memory tree being walked is caller-owned, so the trust boundary
  matches.
