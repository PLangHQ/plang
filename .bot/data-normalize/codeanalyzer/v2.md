## M1 — `json.Writer.Value` STJ fallback — **closed**

`json/writer.cs:147-156` now throws a typed `NormalizeException` with key
`NormalizeUnexpectedLeafType` instead of routing through
`JsonSerializer.Serialize`. The wire-view filter is now closed at the gate:
anything Normalize misses fails loud, and reflection can't sneak back in.
The new typed BCL slots (P3) drain the prior fall-through candidates
(`DateTimeOffset`, `TimeSpan`, `Guid`, `Enum`, `float`) into explicit cases
above the default — so the throw stays reachable only for genuine contract
violations, not for normal payloads.

## M2 — `Properties` slot bypass — **outer site closed, inner site NOT (see V1)**

`Wire.cs:404-415` correctly routes outer-Data Properties values through
`Data.NormalizeValue(kvp.Value, View, …)` + `jsonWriter.Value(...)` — uses
the converter's `View`, symmetric with the outer value slot.

But `json.Writer.EndRecord(record)` at `writer.cs:78-87` hard-codes
`app.View.Out` when doing the same walk for **inner** records. Inner Datas
are emitted inline via `json.Writer.Value` case `@this nested` →
`BeginRecord` → `Value` → `EndRecord`. So the moment a Data graph has a
nested Data anywhere in `Value`, that nested Data's Properties skips the
outer Wire's View entirely.

That's not theoretical — it's the same defect M2 was filed for, at the
second of two emission sites. Easy to miss because the existing test surface
exercises outer Properties (M2 fixture) but no fixture sends a Data-in-Data
with inner Properties through the Store path.

## V1 (major, new) — `json.Writer.EndRecord` hard-codes `View.Out` when normalizing inner-Data Properties

`PLang/app/channels/serializers/json/writer.cs:78-87`

```csharp
var normalized = app.data.@this.NormalizeValue(kvp.Value, app.View.Out, …);
//                                                          ^^^^^^^^^^^^
```

This is M2 at its second emission site. M2's diagnosis was "asymmetric view
discipline at the Properties slot leaks the wire-view filter." The Wire.Write
half is fixed; the json.Writer half still leaks — for inner Datas, only.

**Concrete impact (Store path):**

The plang serializer's `Store()` constructs `Wire(View.Store)` for the
outbound options. Sqlite persistence (`Sqlite.Save` line ~230) routes
through this. When a Data being stored carries a nested Data in its Value
graph (e.g. `GoalCall.Parameters: List<Data>` round-tripped through
identity / permission / settings tables), the outer Wire emits with
View.Store correctly, but the inner Data's Properties get View.Out
treatment from json.Writer.EndRecord. Concretely:

- Inner `[Sensitive]` fields in Properties get **stripped on save** — silent
  data loss, because Out excludes Sensitive while Store keeps it.
- Inner `[Store]`-only fields are **excluded on save** — Out only honors
  `[Out]`, so a `[Store]`-marked field without `[Out]` vanishes.

Whether this corrupts data today depends on whether anyone deposits view-
sensitive content into a nested Data's Properties. I haven't found a
current production callsite that does, but the contract is silently
inconsistent — a future caller will trip it.

**Why no test caught it:** The new M2 fixture
(`CanonicalizationTests.EnsureInnerSigned_RecursesIntoDictionaryValues`)
covers the sign-walk, not the view-pass-through. No fixture exercises
Store-mode + inner-Data-with-Properties round-trip.

Fix sketch:

```csharp
public sealed class Writer : IWriter {
    private readonly app.View _view;
    public Writer(Utf8JsonWriter writer, JsonSerializerOptions? options = null,
                  app.View view = app.View.Out) {
        ...
        _view = view;
    }
    // In EndRecord:
    var normalized = app.data.@this.NormalizeValue(kvp.Value, _view, ...);
}
```

And update Wire.Write's construction site to pass `View`.
