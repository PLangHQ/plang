## M1 ‚Äî `json.Writer.Value` STJ fallback ‚Äî **closed**

`json/writer.cs:147-156` now throws a typed `NormalizeException` with key
`NormalizeUnexpectedLeafType` instead of routing through
`JsonSerializer.Serialize`. The wire-view filter is now closed at the gate:
anything Normalize misses fails loud, and reflection can't sneak back in.
The new typed BCL slots (P3) drain the prior fall-through candidates
(`DateTimeOffset`, `TimeSpan`, `Guid`, `Enum`, `float`) into explicit cases
above the default ‚Äî so the throw stays reachable only for genuine contract
violations, not for normal payloads.

## M2 ‚Äî `Properties` slot bypass ‚Äî **closed (with V1 residual)**

`Wire.cs:404-415` and `json/writer.cs:78-87` both now route Properties values
through `Data.NormalizeValue + jsonWriter.Value`. The asymmetry with the
read side is gone; any domain object deposited in Properties goes through
`Tagged` filtering symmetric with the value slot.

**Residual (V1):** `json.Writer.EndRecord(record)` hard-codes
`app.View.Out` (writer.cs:84) when normalizing Properties values. `Wire.Write`
correctly uses the converter's `View` field (Wire.cs:407). When an inner
nested Data is emitted (`json.Writer.Value` case `@this nested` ‚Üí
`EndRecord`), inner Properties get Out filtering regardless of the outer
view. For Store mode (sqlite persistence), this means inner Datas'
Properties values would lose [Sensitive]/[Store]-only fields if any showed
up there. In practice Properties holds primitives + metadata, so the harm
is theoretical today ‚Äî but the asymmetry is the same shape M2 was filed for.
Two fixes either work:

- Plumb a `View` into `json.Writer`'s constructor, set at the Wire site.
- Inline-emit the inner record's Properties from Wire instead of routing
  through json.Writer.EndRecord (so the only Properties walker is Wire,
  which already knows View).

I'd lean toward the first ‚Äî it keeps Wire writing the outer shape and
json.Writer owning the format details. Polish-only; not a blocker.

## V1 (polish, new) ‚Äî `json.Writer.EndRecord` hard-codes `View.Out` when normalizing inner-Data Properties

See "Residual" under M2 above. Filing as a polish-only follow-up for v3 or
deferred to a future serialization pass. Not a blocker because Properties
in current usage holds primitives + envelope metadata (`VerifiedFlag`,
provenance) rather than domain objects with [Sensitive] fields. If/when
Properties' shape broadens, this asymmetry becomes real.

Fix sketch:

```csharp
public sealed class Writer : IWriter {
    private readonly app.View _view;
    public Writer(Utf8JsonWriter writer, JsonSerializerOptions? options = null,
                  app.View view = app.View.Out) {
        ...
        _view = view;
    }
    // In EndRecord:
    var normalized = app.data.@this.NormalizeValue(kvp.Value, _view, ...);
}
```

And update Wire.Write's construction site to pass `View`.
