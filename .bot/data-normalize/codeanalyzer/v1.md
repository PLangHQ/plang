# codeanalyzer v1 — data-normalize

**Scope:** Stage 1 + 2 + 3 landed (`coder/v1/report.md`). I focused on the new
surfaces: `Data.Normalize`, `Data.Reconstruct`, `Tagged` (the wire-view filter,
née Wire), `IWriter`/`json.Writer`, the `Wire` converter changes, and the
Stage 1 `[Out]`/`[Masked]`/`[Store]` migration. Cross-cutting scans: `System.IO`,
`Console.*`, OBP smells, bare-catch.

**Baseline:** Clean rebuild green (zero PLNG002). C# tests 3379/3379 pass.
PLang tests 233/233 pass. No build regressions found on this pass.

**Findings:** 3 major, 8 polish/observations. None of the criticals on this
branch flow into the live wire path *today* (Stage 2b wiring is still
deferred), but two of them will silently widen the wire-leak surface as soon
as Stage 2b lands. Worth fixing before that wiring goes in.

## M1 (major) — `JsonWriter.Value` default branch falls back to STJ and bypasses every wire-view gate

`PLang/app/channels/serializers/json/writer.cs:135-140`

```csharp
default:
    // Fallback: hand the unexpected object to STJ. Reaching here means
    // Normalize missed a case — surface a typed error in the future.
    JsonSerializer.Serialize(_writer, normalized, _options);
    return;
```

The whole premise of this branch is "reflection fires exactly once, inside
Normalize, never in format encoders." That guarantee is what justifies
`[Out]`-as-positive-whitelist, `[Sensitive]` exclusion, and `[Masked]`
redaction at the boundary. This default-branch fallback re-introduces STJ
reflection at the last gate — and STJ doesn't consult `Tagged`. If anything
slips past Normalize (a future type added to a domain class, a heterogeneous
list element of an unexpected runtime type, a `DateTimeOffset`/`TimeSpan`/
`Guid` value — Normalize keeps those as leaves but `IWriter` has no slot;
see P3), STJ will reflect it and emit every public property unfiltered.

The comment already flags this as TODO ("surface a typed error in the
future"). Make that now, not later. A `case _` that throws a typed
`NormalizeException("NormalizeUnexpectedLeafType", $"{normalized.GetType()}")`
keeps the discipline closed.

**Why now:** without Stage 2b wiring, the Value-slot still routes through
`WireJsonConverter.Write`'s legacy STJ path, so this fallback is dormant.
The moment Stage 2b switches the value slot to `Normalize → JsonWriter`
(`coder/v1/report.md` line ~49), this default branch is on the critical path
and can leak sensitive fields if any domain class is missed in the `[Out]`
inventory.

## M2 (major) — `Wire.Write` emits `Properties` via STJ directly, not through Normalize/JsonWriter

`PLang/app/data/Wire.cs:391-401`

```csharp
if (data.Properties.Count > 0)
{
    writer.WritePropertyName("properties");
    writer.WriteStartObject();
    foreach (var kvp in data.Properties)
    {
        writer.WritePropertyName(kvp.Key);
        JsonSerializer.Serialize(writer, kvp.Value, options);   // STJ reflection here
    }
    writer.WriteEndObject();
}
```

`json.Writer.EndRecord` has the same pattern (line 71-76). Both sidestep
the wire-view filter for the `properties` slot. The read side
(`ReadPropertyPrimitive`, Wire.cs:280-313) is restricted to primitive /
list / dict — so the *contract* of Properties seems to be primitives only.
But the write side accepts arbitrary `object?` and reflects it. Asymmetric:
producers can deposit a sensitive domain object into Properties (no compiler
gate stops them) and the wire will dump every public field via STJ.

Two ways to close the gap:

- **Tighten the type:** make `Properties` a `Dictionary<string, object?>`
  whose setter validates that values are primitives / lists-of-primitives /
  dicts-of-primitives (matching the read shape). Build-time gate.
- **Or route through Normalize:** in `Wire.Write` and `json.Writer.EndRecord`,
  pass each kvp.Value through `Data.NormalizeValue` + `jsonWriter.Value`,
  same as the value slot. Asymmetry gone, no silent reflection.

The Wire converter's own comment (Wire.cs:387-390) acknowledges Properties
sits outside the signature-walk; this finding extends the same observation
to the wire-view filter.

## M3 (major) — `Wire.EnsureInnerSigned` does not recurse into dictionary values

`PLang/app/data/Wire.cs:153-167`

```csharp
private static void EnsureInnerSigned(object? value)
{
    if (value is null) return;
    if (value is @this inner) { ... }
    if (value is System.Collections.IEnumerable seq && value is not string)
    {
        foreach (var item in seq) EnsureInnerSigned(item);
    }
}
```

`IDictionary` is `IEnumerable<DictionaryEntry>` — iterating it yields
`DictionaryEntry` boxes, not the values. `DictionaryEntry` is neither `@this`
nor `IEnumerable`, so the recursion bottoms out immediately. Any inner Data
held as a *value* inside a dictionary inside the outer Data's `Value` graph
will never be sealed before the wire write. The outer signature still binds
the canonical bytes, so tampering with the dict entry is detectable — but
the inner Data ships *unsigned*, which breaks the "every Data in scope gets
sealed before any byte leaves" claim at line 357.

Fix: branch on `IDictionary` ahead of the generic `IEnumerable` check and
iterate `dict.Values` (or yield both key and value through the recursion if
keys can be Datas, which they can't in current usage).

Tests for this surface seem to exercise list-of-Data but not dict-of-Data;
worth a fixture.

## P1 (polish) — `Reconstruct.ReconstructObject` bare-catch conflates "no ctor" with "ctor threw"

`PLang/app/data/this.Reconstruct.cs:158-168`

```csharp
object? instance;
try { instance = Activator.CreateInstance(targetType); }
catch
{
    throw new NormalizeException(
        $"No reconstruction strategy for {targetType.Name}: neither parameterless ctor nor FromNormalized hook found.",
        "NormalizeNoReconstructionStrategy");
}
```

The bare `catch` swallows everything `Activator.CreateInstance` can throw —
`MissingMethodException` (no ctor) is one case, but `TargetInvocationException`
(ctor body threw), `MemberAccessException` (ctor not public), etc. are
others. All collapse into "no reconstruction strategy", which is misleading
when the real failure is "your parameterless ctor threw."

Distinguish: catch `MissingMethodException` → strategy-missing (current Key);
catch the rest → `NormalizeReconstructFailed` with the inner exception
attached. Same shape as the positional-ctor branch at line 146-155 already does.

## P2 (polish) — `Tagged.Compute` uses `IsDefined(..., inherit: true)` but PropertyInfo attribute inheritance does nothing

`PLang/app/channels/serializers/filters/Tagged.cs:57-60, 76, 84-86, 97-98, 107`

```csharp
if (p.IsDefined(typeof(OutAttribute), inherit: true) ...
```

PropertyInfo's attribute inheritance is a runtime quirk: attributes on a
*property declaration* are not inherited from a base property declaration
regardless of the `inherit` flag (see Stage 1's `FilePath`/`HttpPath`
explicit `[Out] override string Scheme` workaround at `path/file/this.cs:50`
and `path/http/this.cs:97`). The `inherit: true` here is dead-cargo —
behavior is identical to `inherit: false` — and reads as if base-class
attributes propagate, which they don't.

Drop to `inherit: false` (no behavior change) and add a one-line comment
above the first call explaining the quirk so a future reader doesn't
re-introduce it.

## P3 (polish) — `IWriter` interface has no slot for `Enum` / `Float` / `DateTimeOffset` / `TimeSpan` / `Guid`, but Normalize keeps them as leaves

`PLang/app/data/this.Normalize.cs:52-66` keeps `DateTimeOffset`, `TimeSpan`,
`Guid`, `Enum`, `float` as tree-native leaves. `json.Writer.Value`
(json/writer.cs:99-141) handles `Enum` (line 111) and `float` (line 106) via
private cases, and silently lets `DateTimeOffset`/`TimeSpan`/`Guid` fall to
the STJ default (M1).

The `IWriter` contract documents the allowed types as "primitive (string,
int, long, double, float, bool, DateTime, decimal), byte[], Data, IEnumerable"
(json/writer.cs:93-95) — but the interface (IWriter.cs:19-55) doesn't expose
`Float`, `Enum`, or any of `DateTimeOffset`/`TimeSpan`/`Guid`. A non-reflection
writer (protobuf, MsgPack) that conforms to `IWriter` has no slot to encode
these values that Normalize actually produces.

Two options:

- **Widen `IWriter`** with explicit slots for these types (Normalize keeps
  them as native leaves; every writer must implement).
- **Normalize them away:** map `Enum` → string (`.ToString()`), `float` →
  `double`, `DateTimeOffset` → string (ISO 8601), `TimeSpan`/`Guid` →
  string. Then `IWriter` stays narrow and writers stay trivial.

Either is fine; the inconsistency between the contract and reality is the
finding. I'd lean toward normalize-them-away — it keeps "every format is a
trivial walker" honest.

## P4 (polish) — `[Out, Store]` on `Data.Value` / `Data.Type` / `Data.Properties` / `Data.Error` / `Data.Success` is dead-code

`PLang/app/data/this.cs:203, 274, 184`; `this.Result.cs:31, 38`;
`this.Transport.cs:30-37`.

The `Wire` JsonConverter handles the Data envelope directly, hand-writing
the canonical `{name, type, value, properties, signature}` shape. It never
consults `Tagged.PropertiesFor(typeof(Data), ...)`. The nested-Data branch
in Normalize (`is @this nested`, this.Normalize.cs:73-87) special-cases
@this *ahead* of `NormalizeObject`, so a Data-in-Value also bypasses
`Tagged`. Net: the `[Out, Store]` annotations on `Data` itself never gate
anything — they're decoration that looks like contract.

Two interpretations:

- **Decoration is intentional documentation** — fine, but add a one-line
  comment above the `[Out, Store]` clusters: *"Documentation only — Data's
  wire shape is handled directly by Wire, never via Tagged."*
- **The annotations are aspirational** — i.e. someone meant to move Data's
  envelope through Tagged eventually. In that case, file a TODO or do it.

Right now a reader can't tell which, and that's the smell.

## P5 (polish) — `Reconstruct<T>` always calls `Normalize()` on entry, even when the value is already a tree

`PLang/app/data/this.Reconstruct.cs:38` — every `Reconstruct<T>` call runs
`Normalize()` first. After a deserialize-from-wire, `Value` is already in
the normalized shape (primitive / list / dict / Data tree). Normalize re-
walks the whole graph, re-runs the visited-set + depth machinery, and
allocates fresh copies.

Idempotent in shape (Normalize at this.Normalize.cs:29 says so), so no
correctness issue — just wasteful and counter to the design goal of
"reflection fires once." A `value-is-already-normalized` guard at the entry
point (e.g. when `Value` is already one of the tree types and contains no
non-tree types reachable) would skip the second walk. Could land later;
flagging now while the surface is fresh.

## P6 (polish) — `Reconstruct`'s `Dictionary<K,V>` branch silently drops entries when key conversion returns null

`PLang/app/data/this.Reconstruct.cs:97-100`

```csharp
var k = AppTypes.ConvertTo(child.Name, keyType);
var v = Walk(child.Value, valueType, ctx);
if (k != null) dictInstance[k] = v;
```

If `AppTypes.ConvertTo` returns null for a non-empty child name (e.g. the
name "abc" being walked into `Dictionary<int, T>`), the entry vanishes
without trace — no exception, no warning. A receiver gets a partial dict
and never knows. Match the rest of Reconstruct's error model: throw a typed
`NormalizeReconstructFailed` instead of a silent drop.

## P7 (polish) — `Normalize.IsTreeLeafType` excludes `DateTimeOffset` / `TimeSpan` / `Guid`

`PLang/app/data/this.Normalize.cs:192-195`

`NormalizeValue` lists `DateTimeOffset`, `TimeSpan`, `Guid` as tree-native
leaves (line 53-54). `IsTreeLeafType` (used by `IsHomogeneousPrimitive`)
doesn't list them. Effect: a homogeneous `List<Guid>` falls into the
heterogeneous-wrapped path instead of the bare-list fast path. Cosmetic
inconsistency — easy to forget when next adding a tree-leaf type. Keep the
two lists in lockstep, ideally one method consumed by both.

## P8 (observation) — Path scheme on the wire is informational only; Reconstruct's hook reads only "relative"

`PLang/app/data/this.Reconstruct.cs:221-256` — the path hook reads the
"relative" child and calls `path.Resolve(relative, ctx)`. The "scheme" child
on the wire (`path/this.cs:18`, marked `[Out, Store]`) is *not consulted on
inbound*. Resolve works because path strings carry their scheme prefix
("http://...", absolute file paths, etc.) — but the scheme field is wire
overhead that isn't load-bearing for reconstruction. Worth a comment so the
next reader doesn't assume it's part of the round-trip contract.

(For pure file paths, this also means an absolute-path-outside-root re-
anchors to whatever root the receiving Context uses. That's the existing
PLang portability invariant, not a regression of this branch.)

## What I didn't find but checked

- **`System.IO.*` in new code** — clean. Normalize/Reconstruct/Tagged/IWriter/
  JsonWriter touch zero filesystem.
- **`Console.*` writes** — clean.
- **Bare-catch / silent-error patterns** — only the M1 default-branch and
  P1 ctor catch surfaced. The `Default.cs:340` catch in builder's plan-dump
  diagnostic is a legitimate "fallback messaging mustn't itself throw"
  catch, comment is clear, original error still surfaces. OK.
- **OBP smells** — the caches in `Tagged._cache`, `_hookCache`,
  `_settablePropsCache` are all class-private with controlled surface and
  test-clear hooks. No outside-iteration smell. `NormalizeException` is a
  proper typed error with a `Key` field — matches the project's existing
  failure-Key discipline.
- **The `MarkOuterForHash` ref-counting AsyncLocal** — looked again because
  security v1 flagged the recursion bomb on the read side. Read-depth
  threshold (MaxReadDepth=64) is in place; write side ref-counting handles
  the chained-Hash scenario. Both paths look correct.

## Verdict

Three majors (M1-M3) live around the wire-emit + sign-walk surface and
should land before Stage 2b switches the value-slot pipeline. P-series are
maintenance / consistency polish; coder's call when (or whether) to absorb.

Tests are green at the baseline so none of the majors are *failing today* —
they're latent defects whose blast radius grows when wiring lands.
