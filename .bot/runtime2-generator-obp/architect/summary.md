# Architect — runtime2-generator-obp

- **v1** ([details](v1/summary.md)) — Design only. Plan to restructure `LazyParamsGenerator` into OBP shape: round 1 extracts the 11-way per-property `if/else` into a polymorphic `ActionProperty` hierarchy under `Emission/Property/`. Markers, `ExecuteAsync`, helpers, and snapshot stay procedural. Hard promise: byte-for-byte identical `.g.cs` output (regression harness pins current output as `golden/`). Plan approved by Ingi; ready for coder handoff.
- **v2** ([plan](v2/plan.md)) — Fresh-eyes review of v1 combined the structural OBP refactor with contract simplification (helpers move to `ActionBase`, runtime kind-switch via `ParamKind`, thin generated `ExecuteAsync`). Built the property hierarchy in its final shape and replaced byte-for-byte golden harness with a runtime test matrix as the regression contract. Superseded by v3 before any code touched.
- **v3** ([plan](v3/plan.md)) — In-conversation re-review dropped `ActionBase` and `ParamKind` entirely. Three architectural shifts: helpers belong on `Action.@this` (data owner), resolution moves to `Data.@this`, property dispatch dissolves to one uniform line per parameter. Hierarchy collapses to two leaves (`DataProperty`, `ProviderProperty`). Build-time validation enforces `Data<T>`-only properties. Approved structurally; superseded by v4 after the runtime2 merge revealed an architectural problem with v3's choice to put resolution on `Data.Value`'s getter.
- **v4** ([plan](v4/plan.md)) — One sharpening: resolution is a read transformation (`Data.As<T>(context)`), not stored state on `Data.Value`. Pulled in runtime2's accumulated work (Builder V2 + ~50 commits) and recognized the recent `_resolved`/`_rawValue`/`ResetResolution`/`IsDeferredActionTemplate` cluster as plaster on the same wound v3 was about to repeat. v4 dissolves all of it: `Data` is stateless w.r.t. resolution; the handler's `__backing` field is the only cache, with the right lifetime by construction. Phase 2 simplifies dramatically; Phases 0/1/3/4/5 keep v3's structure with minor tightening. Awaiting Ingi's read-back.
- **v5** ([plan](v5/plan.md)) — Returns to v4's stated goal of deleting `[VariableName]` after the `runtime2-variablename-migration` branch concluded the original migration path was brittle (bare-name LLM emission `value="x"` silently miswrites under `Data<string>.As<>.Name`). v5 takes the alternate route: introduce a `Variable` record (`Name`, `RawValue`, `WasPercentWrapped`, plus implicit `string` conversion) and declare every former `[VariableName] string` as `Data<Variable>`. Signing lives on the wrapper (`Data<Variable>.Signature`) not on Variable itself — Variable is a value, like `string`. `Variable.Resolve` collapses `%x%` and `x` symmetrically, matching today's `__StripPercent` behavior. Reuses existing `As<T>` dispatch to `static Resolve` at `Data.cs:612-624` — no new wiring. Three-commit phasing: add Variable + tests, migrate 22 handler sites, delete Legacy/. PLNG001 collapses to two rules (`Data<T>` or `[Provider] T`). Awaiting Ingi's read-back before coder handoff.
