## B1 — `Events._active` is `static`: cross-channel recursion guard contamination

**File:** `PLang/App/Channels/Channel/Events/this.cs:22`

```csharp
private static readonly AsyncLocal<HashSet<string>?> _active = new();
```

`_active` is `static`, so all `Channel.Events.@this` instances share the same `AsyncLocal` slot. The recursion guard is meant to be per-channel — "is binding X currently firing on *this* channel." With `static`, the guard is actually per-flow across all channels.

**Concrete failure path:**

1. Channel "output" fires BeforeWrite — binding with ID `"abc"` → `Events.Enter("abc")` adds `"abc"` to the flow's shared HashSet.
2. That handler writes to channel "logger" — `FireBefore` on `logger.Events` calls `Events.IsActive("abc")` → checks the SAME static `_active` HashSet → finds `"abc"` → **skips logger's binding "abc"** (false-positive guard).

Whether this fires in practice depends on binding ID uniqueness. If IDs are per-channel sequential integers (`0, 1, 2 ...`), cross-channel collision is guaranteed on any two-channel scenario. If they're globally unique UUIDs, collision requires the same binding object to be registered on multiple channels — possible via shared handler references.

Either way, the shape is wrong. A per-channel guard must use an instance `AsyncLocal`, not a static one.

**Fix:** Remove `static` from `_active`.

```csharp
// Was: private static readonly AsyncLocal<HashSet<string>?> _active = new();
private readonly AsyncLocal<HashSet<string>?> _active = new();
```

## L1 — `AsyncLocal<HashSet<string>?>` with mutable value: child-task inheritance hazard

**File:** `PLang/App/Channels/Channel/Events/this.cs:69-74`

```csharp
public IDisposable Enter(string bindingId)
{
    var set = _active.Value ??= new HashSet<string>();
    set.Add(bindingId);
    return new Releaser(set, bindingId);
}
```

`AsyncLocal<T>` propagates its **reference** to child tasks for reference types. If `_active.Value` is non-null when a `Task.Run`/`Task.WhenAll` child is spawned, the child inherits the same `HashSet<string>` object. The child's `Enter` then calls `set.Add(...)` on the **parent's** set — and the parent sees the child's bindings as "active." When the child's `Releaser.Dispose()` calls `set.Remove(...)`, it removes from the parent's set, potentially clearing a binding the parent was still using.

This is latent: the current code has no parallel fan-out inside `FireBefore`/`FireAfter` (both iterate sequentially). But any handler that spawns parallel sub-tasks while `Enter` is live will trigger this. Binding handler → `Task.WhenAll(...)` is a plausible future pattern (e.g., parallel fan-out to multiple recipients).

**Fix:** At `Enter`, always assign a **new child-local** HashSet rather than inheriting the parent's:

```csharp
public IDisposable Enter(string bindingId)
{
    var parent = _active.Value;
    var set = parent == null
        ? new HashSet<string> { bindingId }
        : new HashSet<string>(parent) { bindingId };   // copy-on-write
    _active.Value = set;
    return new Releaser(this, bindingId);
}
```

`Releaser` then restores `_active.Value` to the parent set on dispose rather than removing from the set.

## I1 — `Variables.Snapshot()` ignores the overlay

**File:** `PLang/App/Variables/this.cs:689-700`

```csharp
public Dictionary<string, object?> Snapshot()
{
    var dict = new Dictionary<string, object?>(StringComparer.OrdinalIgnoreCase);
    foreach (var kvp in _variables)
    { ... }
    return dict;
}
```

`Snapshot()` only reads `_variables` (the actor-shared dict) and ignores `Calls.Current`. If called inside a fork overlay (e.g., the `GoalMigrationPayload` at `Goal/this.cs:101`) any overlay-only writes are silently absent from the snapshot. Currently limited to the Stage 9 stub (migration is deferred), but `Snapshot()` is general-purpose and will silently drop overlay state whenever called from inside a forked scope.

Not a blocking bug today given the migration stub, but it will surprise when real callers land. Low priority.
