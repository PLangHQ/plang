# Phase 0.4 — Type Preservation Gaps

## Goal
Audit and fix list module handlers that don't properly set `Data.Type` on return values. Ensure all handlers return Data with correct type information so the type-aware pipeline works end-to-end.

## Depends On
- Nothing — independent of other Phase 0 sub-phases

## Context
Most handlers already set types correctly:
- `convert/*` handlers explicitly set types (e.g., `Type.FromName("int")`, `Type.FromMime("application/json")`)
- `variable/set` preserves types from source
- `variable/get` returns stored type
- `file/read` returns file wrapper type

But some `list/*` handlers may not set `Data.Type` on their return Data, breaking the type chain.

## Approach

### 1. Read every list handler file
Check the `Run()` method's return statement. Look for:
- `Data.Ok(someValue)` — missing type parameter → type gets auto-inferred from value's CLR type
- `Data.Ok(someValue, Type.FromName("list"))` — explicit type → correct

### Files to audit (all in `PLang/Runtime2/modules/list/`)
- `add.cs` — returns the modified list
- `remove.cs` — returns the modified list
- `get.cs` — returns a single item (type should match item type)
- `set.cs` — returns the modified list
- `count.cs` — returns int
- `contains.cs` — returns bool
- `indexOf.cs` — returns int
- `first.cs` — returns a single item
- `last.cs` — returns a single item
- `sort.cs` — returns the sorted list
- `reverse.cs` — returns the reversed list
- `unique.cs` — returns the deduplicated list
- `join.cs` — returns string
- `split.cs` — returns list
- `range.cs` — returns new list
- `flatten.cs` — returns flattened list

### 2. Fix pattern
For handlers returning lists, ensure they return:
```csharp
return Data.Ok(result, Type.FromName("list"));
```

For handlers returning scalars (count → int, contains → bool, join → string), verify auto-inference works correctly. If the value is a CLR int/bool/string, Data's constructor will auto-infer the type. Explicitly setting is cleaner but auto-inference should work.

### 3. Also check types.cs wrapper
Read `PLang/Runtime2/modules/list/types.cs` to see if the `list` wrapper type affects type inference. If there's a custom wrapper, the Data.Type auto-inference might infer the wrapper's CLR type rather than "list".
