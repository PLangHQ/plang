# Phase 0.2 — Error Categories (Application vs Runtime)

## Goal
Add an `ErrorCategory` enum and `Category` property to the IError hierarchy. This distinguishes developer-intentional errors (validation, throw) from unexpected system failures (null ref, file not found).

## Depends On
- Phase 0.1 (Data.Fail→Data.Error rename) should be done first, but not strictly required

## Design

### New enum: `PLang/Runtime2/Errors/ErrorCategory.cs`
```csharp
namespace PLang.Runtime2.Errors;

/// <summary>
/// Classifies the origin/intent of an error.
/// </summary>
public enum ErrorCategory
{
    /// <summary>
    /// Developer intentionally created this error (throw error, validate, business rule).
    /// The PLang developer chose to raise this — it's part of the application logic.
    /// </summary>
    Application,

    /// <summary>
    /// Unexpected system/runtime failure (file not found, null reference, timeout, handler crash).
    /// The PLang developer did not anticipate this error.
    /// </summary>
    Runtime

    // Future: External — errors from external services (HTTP 500 from an API, DB connection failure)
}
```

### Category mapping by error type

| Error Type | File | Category | Reasoning |
|-----------|------|----------|-----------|
| `Error` (base) | `Errors/Error.cs` | `StatusCode < 500 ? Application : Runtime` | Convention: 4xx = app, 5xx = runtime |
| `ValidationError` | `Errors/ValidationError.cs` | `Application` (always) | Input validation = intentional |
| `AssertionError` | `Errors/AssertionError.cs` | `Application` (always) | Test assertions = developer feedback |
| `ActionError` | `Errors/ActionError.cs` | Inherit from base | Follows StatusCode convention |
| `StepError` | `Errors/StepError.cs` | `Runtime` (always) | Unexpected runtime failure |
| `GoalError` | `Errors/GoalError.cs` | `Runtime` (always) | System-level (goal not found, cancelled) |
| `ServiceError` | `Errors/ServiceError.cs` | `Runtime` (always) | Internal service failure |

## Changes

### 1. Create `PLang/Runtime2/Errors/ErrorCategory.cs`
New file with the enum (see above).

### 2. Add to `PLang/Runtime2/Errors/IError.cs`
Add property:
```csharp
ErrorCategory Category { get; }
```

### 3. Add default to `PLang/Runtime2/Errors/Error.cs`
```csharp
public virtual ErrorCategory Category => StatusCode < 500 ? ErrorCategory.Application : ErrorCategory.Runtime;
```

### 4. Override in subclasses
- `ValidationError.cs` — add: `public override ErrorCategory Category => ErrorCategory.Application;`
- `AssertionError.cs` — add: `public override ErrorCategory Category => ErrorCategory.Application;`
- `StepError.cs` — add: `public override ErrorCategory Category => ErrorCategory.Runtime;`
- `GoalError.cs` — add: `public override ErrorCategory Category => ErrorCategory.Runtime;`
- `ServiceError.cs` — add: `public override ErrorCategory Category => ErrorCategory.Runtime;`

Note: `ActionError` inherits from `Error`, so it gets the StatusCode-based default.

### 5. Improve `Error.Format()` in `PLang/Runtime2/Errors/Error.cs`
Differentiate output based on Category:

**Application errors** — concise, developer-friendly:
```
⚠️  Error(400) — validation
Error: Email address is required
📝 Step: - validate %email% is not empty
📄 File: \Users\Start.goal:5
```

**Runtime errors** — full detail with stack trace and C# exception:
```
🔴  System Error(500) — Exception
...existing full format with call stack, code snippet, fix suggestions, C# developer info...
```

The key change: Application errors skip the call stack, C# exception, and verbose formatting sections. They show just message + step + file.

### 6. Write tests: `PLang.Tests/Runtime2/Errors/ErrorCategoryTests.cs`
- `Error_4xx_IsApplication()` — Error with statusCode 400 → Category == Application
- `Error_5xx_IsRuntime()` — Error with statusCode 500 → Category == Runtime
- `ValidationError_AlwaysApplication()` — regardless of status code
- `AssertionError_AlwaysApplication()`
- `StepError_AlwaysRuntime()`
- `GoalError_AlwaysRuntime()`
- `ServiceError_AlwaysRuntime()`
- `ApplicationError_FormatIsConcise()` — Format() output is short
- `RuntimeError_FormatHasFullDetail()` — Format() output includes stack trace
