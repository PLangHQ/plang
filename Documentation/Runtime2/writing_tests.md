# Writing Tests for PLang Runtime2

Guide for writing both C# unit tests and PLang integration tests.

---

## Two Types of Tests — Both Required

| Type | What it validates | Location |
|------|------------------|----------|
| **C# unit tests** | Handler logic in isolation (no LLM, no builder) | `PLang.Tests/Runtime2/Modules/<module>/` |
| **PLang .goal tests** | Full pipeline: LLM builder → .pr generation → GoalMapper → runtime execution | `Tests/Runtime2/<ModuleName>/` |

C# tests run fast and catch logic bugs. PLang tests catch builder misunderstandings and mapping errors.

---

## PLang Test Goals (*.test.goal)

### Rules

1. **Goal name MUST be `Start`** — the test runner (`plang !test`) looks for a goal named `Start` in each `.test.pr` file. Any other name causes `"Goal 'Start' not found"`.

2. **One test suite per directory** — each `Tests/Runtime2/<ModuleName>/` folder has one `<Name>.test.goal` and optionally supporting `.goal` files.

3. **Supporting goals go in separate files** — if a test needs to call another goal (e.g., a mock handler or a loop body), put it in its own `.goal` file in the same directory.

4. **Use `assert` for verification** — the test runner tracks `AssertionError` failures per step.

### File Structure Example

```
Tests/Runtime2/Mock/
  Mock.test.goal          ← main test (goal named "Start")
  HandleMockPost.goal     ← supporting goal called by the test
  .build/
    mock.test.pr          ← generated by builder
    handlemockpost.pr     ← generated by builder
```

### Minimal Test Template

```plang
Start
- set %value% = "hello"
- assert %value% equals "hello", "value should be set"
```

### Build and Run Workflow

```bash
# 1. Build the C# project first (if handlers changed)
dotnet build PlangConsole/PlangConsole.csproj

# 2. Build PLang tests (from Tests/Runtime2 directory)
cd Tests/Runtime2
plang build

# 3. ALWAYS read the generated .pr file and verify correctness
#    Check: module name, action name, parameter names/values/types, return variables

# 4. Run a single test
plang "Mock/Mock.test.goal"

# 5. Run all tests
plang !test
```

### After Building — Always Verify the .pr File

This is the most critical step. The LLM builder can misunderstand your intent. After `plang build`, read the generated `.pr` file and check:

- **Module name** — is it the correct module? (e.g., `mock` not `event`)
- **Action name** — is it the correct action? (e.g., `intercept` not `mock`)
- **Parameter names** — do they match the handler's property names?
- **Parameter values** — are literals correct? Are `%var%` references preserved?
- **Parameter types** — reasonable types for the values?
- **Return variables** — does the `return` array have the right `%variable%` name?

If the .pr file is wrong, the step will fail at runtime with confusing errors. Diagnosing .pr correctness is always step one.

### Common .pr Issues

| Symptom | Likely Cause |
|---------|-------------|
| `ActionNotFound` error during build | LLM mapped to wrong module.action (e.g., `mock.mock` instead of `mock.intercept`) |
| `Goal 'Start' not found` | Goal name in .goal file isn't `Start` |
| Step runs but assertion fails with wrong value | Parameter value in .pr is wrong — check the .pr file |
| `Built .pr file not found` | Run `plang build` first |
| Build says "approved" but .pr has old data | Content hash matched — delete .pr and rebuild |

## Writing Step Text the Builder Understands

The LLM builder maps natural language steps to `module.action` calls. These tips help it map correctly:

### Use Explicit Module.Action Phrasing

The builder works best when the step text closely mirrors the action summary format:

```plang
/ Good — module and action are clear
- mock intercept "file.read", return "mocked content", write to %mockRead%
- assert %value% equals 42
- convert "hello" to base64, write to %encoded%
- foreach %items%, call ProcessItem item=%item%

/ Bad — ambiguous, builder may guess wrong
- mock file.read         ← builder may generate mock.mock instead of mock.intercept
- check value is 42      ← builder may not find assert.equals
```

### Name Avoidance

Avoid action names that match the module name. The LLM confuses `mock.action` (module=mock, action=action) with `mock.mock`. This is why we renamed `mock.action` to `mock.intercept`.

### Variable References

- Variables use `%name%` syntax: `%user%`, `%user.name%`, `%items[0]%`
- Return variables: `write to %varName%`
- Object literals: `{name: "john", age: 20}`
- Array literals: `["a", "b", "c"]`
- Comments use `/` prefix: `/ This is a comment`

### Assertions Available

```plang
- assert %actual% equals %expected%, "optional message"
- assert %actual% not equals %other%
- assert %value% is true
- assert %value% is false
- assert %value% is null
- assert %value% is not null
- assert %container% contains %element%
- assert %a% greater than %b%
- assert %a% less than %b%
```

### Mocking Available

```plang
/ Return value mock — skip real action, return fixed value
- mock intercept "file.read", return "mocked content", write to %mockRead%

/ Spy — track calls but let real action run
- mock intercept "output.write", write to %spyWrite%

/ Verify call count
- mock verify %mockRead% called 2 times

/ Access mock data via variable navigation
- assert %mockRead.callCount% equals 1
- assert %mockRead.calls[0].parameters.path% equals "config.json"
- assert %mockRead.isSpy% equals false
```

## C# Unit Tests

### Test Framework

PLang uses **TUnit** (not xUnit/NUnit). Key differences:

- Run: `dotnet run --project PLang.Tests` (not `dotnet test`)
- Assertions: `await Assert.That(value).IsTrue()` (fluent async style)
- Attributes: `[Test]` (not `[Fact]` or `[TestMethod]`)

### Test Setup Pattern

Handlers need a `PLangContext` to work. Use this pattern:

```csharp
private (PLangContext context, MemoryStack memory, Engine engine) CreateContext()
{
    var appContext = new PLangAppContext("/app");
    var memory = new MemoryStack();
    var context = new PLangContext(appContext, memory);
    var engine = new Engine(appContext);
    context.RegisterContextVariables(engine);
    return (context, memory, engine);
}
```

### Testing a Handler Directly

Handlers implement `IContext` and have a `Run()` method:

```csharp
[Test]
public async Task Equals_SameInts_Passes()
{
    var (context, _, _) = CreateContext();
    var action = new AssertEquals
    {
        Context = context,
        Expected = 42,
        Actual = 42
    };
    var result = await action.Run();
    await Assert.That(result.Success).IsTrue();
}
```

### Testing Failure Cases

Check for specific error types:

```csharp
[Test]
public async Task Verify_WrongCount_Fails()
{
    var (context, _, _) = CreateContext();
    var handle = new MockHandle { Id = "test", ActionPattern = "file.read" };
    handle.RecordCall(new Dictionary<string, object?> { ["path"] = "test.txt" });

    var verify = new Verify
    {
        Context = context,
        Mock = handle,
        ExpectedCount = 3  // actual is 1
    };

    var result = await verify.Run();
    await Assert.That(result.Success).IsFalse();
    await Assert.That(result.Error is AssertionError).IsTrue();
}
```

### Type Ambiguity

In test files, use `System.Type?` instead of `Type?` to avoid ambiguity with `PLang.Runtime2.Memory.Type`.

### Source Generator Note

The source generator only runs on the PLang project, not PLang.Tests. Test code creates handler instances directly without the generated `ICodeGenerated` wrappers.

## Workflow: NEVER Change .goal Steps When Tests Fail

When a PLang test fails at runtime:

1. **Read the .pr file** — is the builder mapping correct?
2. **If .pr is wrong** — the builder misunderstood the step. Try rephrasing the step text, or investigate if the builder prompt needs improvement. Ask the user before changing steps.
3. **If .pr is correct** — the runtime handler has a bug. Fix the C# handler code.
4. **NEVER silently change PLang test steps** to work around builder or runtime issues. The test steps represent the user's intent — if they don't work, it means the system needs improvement.

## Test Runner Internals

- `plang !test` runs `TestMode.RunAsync()` in `PLang/Runtime2/Core/TestMode.cs`
- Discovers `*.test.goal` files recursively from engine root
- Each test gets a **fresh Engine** — full isolation (no leaked events, variables, or goals)
- Derives .pr path: `<dir>/.build/<name>.test.pr` (e.g., `Mock/.build/mock.test.pr`)
- Runs goal named `Start` from the .pr file
- Tracks `AssertionError` failures via an AfterStep event binding
- Non-assertion errors (runtime crashes) are reported as test errors
- Exit code 0 = all pass, 1 = any failure

## Known Weak Assertions (HTTP Module)

The following tests have weak assertions that the auditor flagged. They pass but don't fully verify the code path. Strengthen when revisiting HTTP tests:

1. **`Stream_Plang_InvalidSignature_SetsErrorOnMemoryStack`** — only checks `!data` is not null. Should verify it's an error Data with a signature-related key. Hard to assert precisely because the streaming callback (goal not found) interacts with the verification error flow.
2. **`Get_PlangResponseInvalidSignature_ReturnsError`** — checks `Error.Key` is not empty but doesn't assert a specific key. The key depends on how the signing module reports the error (could be `SignatureInvalid`, `InvalidType`, etc. depending on which verification step fails first on a bogus signature).

These are tracked as technical debt, not blocking issues. The non-streaming signing tests (`Get_Signed_XSignatureIsValidJson`, `Get_SignedPlangResponse_SetsServiceIdentity`) have strong assertions and cover the happy path thoroughly.
